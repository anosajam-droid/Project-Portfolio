Building the information framework before anyone touches the model

Most BIM problems on site trace back to decisions not made early enough. The BIM Execution Plan is where those decisions get made. I authored a post-appointment BEP under ISO 19650-2:2018 for a 20-storey mixed-use development — defining how every piece of information would be created, shared, and owned across the full project lifecycle.

ISO 19650-2:2018 Post-Appointment BEP Astria Mixed Development · Oxford Client: Horizon Developers 24-month programme
The Project

A 20-storey mixed-use scheme that needed a digital delivery strategy

Astria Mixed Development — a high-density scheme on John Garne Way, Marston, Oxford — comprises a 20-storey residential tower, a 5-storey commercial building, luxury apartments, and recreational amenities across 10 acres. Client: Horizon Developers. Duration: 24 months.

At post-appointment stage, the delivery team is confirmed but nothing has been modelled yet. This is the moment to define how the project will be digitally delivered — what information is needed, when, in what format, and by whom. Get this right and the project runs on clear information. Get it wrong and you spend the rest of the project chasing correct data from the wrong people.

What I Produced

A BEP that covers every stage from programming to operation

The BEP follows the four-step planning procedure: identify BIM uses, define the execution process, develop information exchanges, and define supporting infrastructure. Each step produced a concrete deliverable — not just a written plan.

The Exchange Information Requirements matrix specifies what software, what native format (.RVT), what open format (IFC), and at what Plan of Work stage each model type must be delivered. The TIDP allocates responsibility per task team. The MIDP gives the client a single view of all information delivery across the project. File naming, CDE configuration, revision codes — all defined before the first model element is placed.

What Was Produced

Six core BEP deliverables

Each one traceable to a specific ISO 19650-2 clause — not just described, but structured and evidenced.

Clause 5.1.2
Project Information Requirements
Defined what the client needs to know at each decision point — from programming through to operation. Establishes the information purpose before any modelling begins.
Output: PIR register with named decision points
Clause 5.4.2
Responsibility Matrix
Five organisations, five named individuals, five contract numbers. Each BIM use allocated to a team lead with estimated effort. RACI framework applied across all information management activities.
Output: Named RACI matrix with contract references
Clause 5.4.3
Exchange Information Requirements
Eight model types — structural, MEP, architectural, HVAC, space planning, site context, existing buildings, model aggregation — each with software, native format, open format, and delivery stage requirements.
Output: EIR matrix across 5 Plan of Work stages
Clause 5.4.4 & 5.4.5
TIDP & MIDP
Task Information Delivery Plans produced per team. Master MIDP compiled to give the client a project-wide view of what information is being delivered, when, and by whom.
Output: TIDP + MIDP with file identifiers and milestones
Clause 5.1.7
File Naming & CDE
Eight-field naming convention established. CDE configured to assign revision codes, status codes, and classification codes to every container. Example: ABC123-IBL-A1-01-M-A-0001_BLDG1.
Output: Naming protocol + CDE configuration spec
Multiple Standards
9-Standard Compliance Framework
ISO 19650 series, ISO 7200, ISO 55000, ISO 4157, CEN/EN 17412-1, BS 5606, BS 8210, BS 8536, and UK National Annex — each applied to a specific aspect of information management.
Output: Standards matrix with application notes
